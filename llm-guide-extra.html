Token 成本到底贵在哪

为什么云端 AI 不是"只有电费"，也不是"边际成本递减"——一份反直觉的成本结构拆解。

01 云端跑大模型，吃哪些资源

显存 (GPU VRAM)

模型参数必须全部装进显存。Llama 70B (FP16) ≈ 140GB，单张 H100 才 80GB，得多卡并联。

显存带宽

每生成 1 个 token，要把整个模型权重从显存读一遍。带宽决定吐字速度。

KV Cache

对话越长，缓存越大。100K token 长对话可能吃掉几十 GB 额外显存——这是长上下文贵的原因。

电力 / 散热 / 网络

一张 H100 满载 700W；数据中心 PUE 约 1.3；液冷、变电站、光纤——基建无底洞。

02 为什么按 Token 收费

关键事实：生成是逐字进行的。大模型不是一次性吐答案，而是一个 token 一个 token 蹦。每蹦一个：把 100GB+ 权重过一遍 → 重新算注意力 → 占用 GPU 几十毫秒。

生成 1000 字回答 ≈ 做 1000 次完整模型前向计算。Token 是"消耗了多少资源"最公平的颗粒度。

Input vs Output 价格不同。输入可以并行处理 → 便宜；输出必须串行一个个吐 → 贵 3–5 倍。
例：Claude Sonnet 输入 $3/百万，输出 $15/百万 token。

上下文是平方级成本。注意力机制 O(n²)：上下文翻倍 → 计算量 4 倍。所以 200K 上下文比 10K 贵得不成比例。

03 为什么没有"边际成本递减"

传统软件多卖一份接近零成本（复制比特）。大模型不一样：

业务边际成本原因
卖软件≈ 0复制比特免费
卖视频 / 搜索很低一份内容 / 索引服务千万人
卖 AI Token近线性每次都现算，无法缓存复用

搜索引擎能免费给你用，是因为一亿人查"天气"用的是同一份索引。AI 给你的每段话都是独一无二的现场计算，没法预存。

例外：Prompt Caching（缓存命中折扣 ~90%）——重复前缀能跳过重算，这部分确实有了边际递减效应。

04 不只是电费——成本拆解

一张 H100 满载 0.7 度电/小时 ≈ $0.07，加散热 ≈ $0.09/小时。但总持有成本：

折旧 (3 年摊销 $30k)$1.00 / 小时
电 + 冷$0.09 / 小时
数据中心摊销 + 运维$0.30 / 小时
资金成本（5–8%/年）$0.20 / 小时
合计≈ $1.6 / 小时

电费只占总成本约 5–6%。真正贵的是硬件折旧（H100 3–4 年就过时）+ 训练研发摊销 + 资金成本。

05 价格构成与未来趋势

当前 token 价格里大致：

  • ~5% 电费
  • ~30% 硬件折旧
  • ~40% 训练 / 研发摊销（GPT-5 训练据传花了几十亿）
  • ~25% 利润 + 风险溢价

过去 2 年同等智能水平的 token 价格降了 ~100 倍：GPT-4 ($30) → GPT-4o ($2.5) → Haiku/Flash ($0.25–1) → 开源本地跑 (~$0)。

终局：5–10 年后主流模型 token 成本可能真的逼近"电费 + 微薄折旧"；但前沿模型永远贵——总有人愿意为最聪明的那个付溢价。

🤖 Agent 实战

Agent 是什么，个人如何开发

从"对话机器人"到"会自己用工具的同事"——Agent 的本质、最小可跑实现、进阶方向。

01 一句话定义

Agent = LLM + 工具 + 循环。模型自己决定调什么工具、读什么结果、下一步做什么，直到任务完成。

区别在于：

  • 普通对话：你问一句，它答一句
  • RAG：检索一次知识库后回答
  • Workflow：开发者预先编排好的步骤链
  • Agent：模型自主规划路径、自主停止

02 四个核心组件

① 模型 (Brain)

规划、决策、生成。Claude Sonnet / GPT / Gemini 都可以。

② 工具 (Tools)

读文件、调 API、跑代码、查数据库——任何"动作"。

③ 记忆 (Memory)

对话历史、向量库、长期 notes 文件。

④ 循环 (Loop)

"思考 → 调工具 → 看结果 → 再思考"，直到模型说"完成"。

03 最小可跑实现（Python）

用 Anthropic SDK，约 50 行就能跑出一个会查天气和算数的 Agent：

import anthropic, json

client = anthropic.Anthropic()

# 1. 定义工具
tools = [
  {
    "name": "get_weather",
    "description": "查询某城市当前天气",
    "input_schema": {
      "type": "object",
      "properties": {"city": {"type": "string"}},
      "required": ["city"]
    }
  },
  {
    "name": "calculator",
    "description": "计算数学表达式",
    "input_schema": {
      "type": "object",
      "properties": {"expr": {"type": "string"}},
      "required": ["expr"]
    }
  }
]

# 2. 工具的真实执行函数
def run_tool(name, args):
    if name == "get_weather":
        return f"{args['city']}: 18°C 多云"
    if name == "calculator":
        return str(eval(args["expr"]))

# 3. 主循环
messages = [{"role": "user", "content": "北京气温的两倍是多少度？"}]

while True:
    resp = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        tools=tools,
        messages=messages,
    )
    messages.append({"role": "assistant", "content": resp.content})

    if resp.stop_reason == "end_turn":
        print(resp.content[0].text); break

    # 4. 模型要求调工具 → 执行 → 把结果喂回去
    tool_results = []
    for block in resp.content:
        if block.type == "tool_use":
            result = run_tool(block.name, block.input)
            tool_results.append({
              "type": "tool_result",
              "tool_use_id": block.id,
              "content": result
            })
    messages.append({"role": "user", "content": tool_results})

就这样——一个 Agent 的"骨架"已经完整：定义工具 → while 循环 → 模型调工具 → 把结果喂回去 → 直到 end_turn。

04 个人开发的进阶清单

稳定性

  • 限制最大循环次数（防止死循环烧 token）
  • 工具失败时返回错误信息而不是抛异常——让模型有机会重试
  • 工具输出加长度上限，避免 context 爆炸

效果

  • System prompt 写清楚"什么时候该用什么工具"——比工具描述更影响成功率
  • 关键路径加 few-shot 示例
  • 开 extended thinking 让模型先想再动

规模化

  • 子 Agent / 多 Agent 协作（一个规划、一个执行、一个验证）
  • 持久化 memory（向量库 / 文件 / SQLite）
  • 人工 checkpoint：高风险动作前必须人确认

05 不用从零写 — 可用的脚手架

  • Claude Agent SDK（Anthropic 官方）— 自带工具循环、文件系统、bash、子 Agent
  • LangGraph — 状态机式 Agent 编排，适合复杂分支
  • OpenAI Agents SDK — 多 Agent handoff 模式
  • Mastra / CrewAI — TypeScript / Python 生态
建议：第一个 Agent 用上面的 ~50 行原生写一遍——理解 tool use 循环的本质比直接套框架更重要。第二个项目再上脚手架。
🛠️ Skills 开发

Skills 是什么，个人如何开发

把"做某件事的最佳实践"打包成模型能按需加载的能力——比 Prompt 工程更工程化的复用方式。

01 Skills 解决什么问题

过去你想让 Claude 学会"按公司规范写周报"，要么：

  • 每次粘贴一大段 system prompt（占 context、容易忘）
  • 微调一个模型（贵、不灵活）

Skill 是第三条路：把这套规范写成一个文件夹，模型在需要时自己读进来用，不需要时不占 context。

类比：Skill 之于 LLM ≈ 函数库之于程序员。你不会把所有库都 import 进每个文件——用到再 import。Skill 也一样。

02 Skill 的物理结构

一个 Skill 就是一个目录，至少包含一个 SKILL.md：

my-skill/
├── SKILL.md          # 必需：描述 + 触发条件 + 操作指引
├── reference.md      # 可选：详细参考（按需读）
├── template.html     # 可选：模板文件
└── scripts/
    └── helper.py     # 可选：工具脚本

SKILL.md 头部用 YAML frontmatter 声明元数据：

---
name: weekly-report
description: 按公司规范生成周报。当用户说"写周报""周总结"
             "本周工作汇报"时使用。
---

# 周报生成指引

## 触发条件
当用户要求生成周报、周总结、周工作汇报时使用。

## 输出格式
1. **本周完成**（3–5 条，每条带数据）
2. **下周计划**（按优先级排序）
3. **风险与求助**（明确写出阻塞）

## 写作要求
- 每条工作必须带可量化指标（数字、百分比、链接）
- 避免"推进""跟进""协调"等模糊动词
- 详细规范见 reference.md

03 模型怎么"知道"该用哪个 Skill

关键机制叫 Progressive Disclosure（渐进式暴露）：

  1. 启动时，只把所有 SKILL.md 的 name + description 加载进 context（每个 Skill 几十 token）
  2. 模型根据用户请求判断"哪个 Skill 的 description 匹配"
  3. 匹配到再读完整的 SKILL.md 正文
  4. 需要细节时再读 reference.md 等附件

所以 description 写得好不好，决定 Skill 会不会被触发。必须包含触发场景关键词（"当用户说 X / 看到 Y 文件 / 需要做 Z 时使用"）。

04 个人开发流程（Claude Code 为例）

Step 1 — 选位置

  • 个人全局：~/.claude/skills/my-skill/ 所有项目可用
  • 项目本地：.claude/skills/my-skill/ 跟着 git 走，团队共享

Step 2 — 写 SKILL.md

  • name 用 kebab-case，全局唯一
  • description 第一句说"做什么"，第二句说"什么时候用"
  • 正文按"前置条件 → 步骤 → 输出格式 → 边界情况"组织

Step 3 — 拆分大文件

SKILL.md 控制在 500 行以内。详细 API 参考、长示例、模板放到独立文件，正文里写"详见 reference.md"。模型按需打开。

Step 4 — 测试

  • 新开会话，输入触发场景的话，看模型是否自动调用
  • 没触发 → 改 description 加更多关键词
  • 触发了但效果差 → 改正文，加更多约束和示例

Step 5 — 分享

打包成 git 仓库即可。Anthropic 也有官方 anthropic-skills 仓库收录优质 Skill（pdf、xlsx、pptx、brand-guidelines 等）。

05 一个真实可用的最小 Skill

新建 ~/.claude/skills/commit-message/SKILL.md：

---
name: commit-message
description: 生成 Conventional Commits 规范的提交信息。
             当用户要求"写 commit message""生成提交信息"
             或运行 git diff 后想提交时使用。
---

# Conventional Commits 生成指引

## 格式
<type>(<scope>): <subject>

<body>

## type 取值
feat / fix / docs / style / refactor / perf / test / chore

## 规则
1. subject 用祈使句，小写开头，不超过 50 字符
2. 重点写"为什么"，不是"做了什么"——diff 已经能看出 what
3. break change 在 footer 加 BREAKING CHANGE: 前缀
4. 不写 "update code" "fix bug" 这种空话

## 流程
1. 先 git diff 看改动
2. 判断主要变更类型
3. 用一句话总结意图
4. 输出符合格式的 commit message

放好后下次说"帮我写个 commit message"，Claude 就会自动遵循这套规范。

06 Skills vs Agents vs MCP — 别搞混

概念本质解决什么
Prompt一段文字告诉模型当下做什么
Skill可按需加载的知识包"该怎么做某件事"的最佳实践复用
Tool / MCP可执行的动作接口让模型连接外部系统（数据库、API、文件）
Agent模型 + 工具 + 循环自主完成多步任务的实体

一个完整的 Agent 系统 = 模型（大脑）+ MCP/Tool（手脚）+ Skills（专业知识手册）+ Loop（行动力）。
